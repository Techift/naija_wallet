## 🔴 What actually blocked you (toolchain mismatch)

### [x] 1. `flutter pub get` fails — project requires a newer SDK than installed  ✅ FIXED
- **Where:** [pubspec.yaml](../pubspec.yaml)
- **What happened:**
  ```text
  The current Dart SDK version is 3.8.1.
  Because naija_wallet requires SDK version ^3.11.0, version solving failed.
  ```
  The pubspec demanded `sdk: ^3.11.0` / `flutter: ^3.41.0`, and `go_router: ^17.2.3` +
  `shared_preferences: ^2.5.5` both need Dart ≥ 3.9 — but you're on Dart 3.8.1. So the
  project could not even resolve dependencies, let alone run.
- **What was done (to stay on your installed Flutter):**
  ```yaml
  environment:
    sdk: ^3.8.0          # was ^3.11.0
    flutter: ">=3.32.0"  # was ^3.41.0
  # ...
  go_router: ^14.0.0          # was ^17.2.3   (17 needs Dart >=3.9)
  shared_preferences: ^2.3.0  # was ^2.5.5    (2.5.4+ needs Dart >=3.9)
  very_good_analysis: ^9.0.0  # was ^10.2.0   (10 needs Dart >=3.9)
  ```
  The go_router APIs you use (`GoRouter`, `GoRoute`, `redirect`) are unchanged across 14–17,
  so your routing code works identically.
- **Alternative (cleaner long-term):** run `flutter upgrade` to a Flutter that ships
  Dart ≥ 3.11, then revert the lines above. Either path is valid; this one keeps your
  current Flutter install.

### [x] 2. iOS won't compile: `Cannot find type 'FlutterSceneDelegate' in scope`  ✅ FIXED
- **Where:** `ios/Runner/SceneDelegate.swift`, `AppDelegate.swift`, `Info.plist`, `project.pbxproj`
- **What happened:** the iOS shell was generated by a newer Flutter (≥ 3.35) that uses the new
  scene-based lifecycle (`FlutterSceneDelegate`, `FlutterImplicitEngineDelegate`). Flutter
  3.32.8 doesn't define those types, so the native build failed.
- **What was done — downgraded the iOS shell to the classic lifecycle:**
  1. Rewrote `AppDelegate.swift` to register plugins in `didFinishLaunchingWithOptions`.
  2. Deleted `SceneDelegate.swift`.
  3. Removed the `UIApplicationSceneManifest` block from `Info.plist`.
  4. Removed the 4 `SceneDelegate.swift` references from `project.pbxproj`.
- **Verified:** `flutter build ios --debug --simulator --flavor development …` →
  `✓ Built build/ios/iphonesimulator/Runner.app`.
- **⚠️ Caveat:** if you later upgrade Flutter to ≥ 3.35, regenerate the iOS shell
  (`flutter create --platforms=ios .`) — the newer Flutter expects the SceneDelegate back.

### [x] 3. Always pass the flavor when running  ✅ (no code change — usage note)
- This project uses flavors, so there is **no plain `Debug`/`Release`** Xcode config — only
  `Debug-development`, etc. Running without a flavor fails with *"Flutter expects a build
  configuration named Debug or similar."*
- **Run it like this** (or pick **"Launch development"** in the VS Code Run dropdown):
  ```bash
  flutter run --flavor development -t lib/main_development.dart -d "iPhone 17 Pro Max"
  ```

## 🟢 The splash-screen "freeze" — explained

Your reported symptom was: *the splash either doesn't show, or shows and never moves to login.*

**Root cause:** it wasn't a logic bug — the app couldn't get a clean run at all because of
the toolchain blockers above (#1, #2). On a version where it couldn't build/launch properly,
the splash was the only thing that ever painted, so it *looked* frozen.

Your actual splash logic on this branch is already correct:
- [splash_screen.dart](../lib/features/splash/view/splash_screen.dart) waits 2s, checks
  `if (!mounted) return;` (good — avoids navigating after disposal), then `context.go(Routes.login)`.
- [app_router.dart](../lib/core/router/app_router.dart) redirect logic is sound and uses the
  `Routes` constants consistently.

Once #1 and #2 were fixed, the flow runs end-to-end: **splash → login → (Login) → home**.
